# ADR-021: キャッシュ基盤として Redis を導入

## ステータス

Proposed

## コンテキスト

パフォーマンス改善のため、読み取り頻度が高くデータ変更が少ないエンドポイントにキャッシュを導入する必要がある。

キャッシュ導入にあたり、以下の要件がある:

- Railway 上で運用可能なこと
- 複数インスタンス間でキャッシュを共有できること（将来のスケールアウトに対応）
- TTL ベースの自動キャッシュ失効をサポートすること
- Go エコシステムで十分な実績があるクライアントライブラリが存在すること

### キャッシュレイヤーの方針

レスポンス DTO には GCS の Signed URL（有効期限 1 時間）が含まれるため、**Repository レベル（DB クエリ結果）** でキャッシュする。
これにより Signed URL の有効期限とキャッシュ TTL の不整合を回避する。

### キャッシュ対象（Phase 1）

| 対象 | キャッシュキー | TTL | 理由 |
|------|--------------|-----|------|
| カテゴリ一覧 | `categories:active` | 1 時間 | 公開 API、高頻度、2 クエリ（一覧 + 集計 JOIN）、データほぼ不変 |
| カテゴリ統計 | `categories:stats` | 1 時間 | カテゴリ一覧・詳細の両方で使用、集計 JOIN が重い |
| カテゴリ詳細 | `category:slug:{slug}` | 1 時間 | 公開 API、データほぼ不変 |
| ボイス一覧 | `voices:all` | 2 時間 | 認証 API、エピソード作成時に毎回呼ばれる、データほぼ不変 |
| ボイス詳細 | `voice:{id}` | 2 時間 | バリデーション時の繰り返し参照、不変データ |

### 将来の拡張候補（Phase 2 以降）

- チャンネル詳細（公開）、エピソード一覧、おすすめ（未認証）、検索結果

## 決定

**Redis** をキャッシュストアとして採用し、Go クライアントには **go-redis/v9** を使用する。

キャッシュ戦略は **Cache-Aside（Lazy Loading）** パターンを採用する:

1. リクエスト時にキャッシュを参照
2. キャッシュヒット → キャッシュから返却
3. キャッシュミス → DB から取得 → キャッシュに書き込み → 返却

キャッシュの無効化は **TTL ベース** を基本とする。
Phase 1 の対象はデータ変更頻度が非常に低いため、書き込み時の明示的な無効化は行わない。

Redis が利用できない場合は DB にフォールバックし、アプリケーションの可用性に影響を与えない設計とする。

## 選択肢

### 選択肢 1: Redis（外部キャッシュストア）

- **メリット**
  - 複数インスタンス間でキャッシュを共有できる
  - Railway で簡単にプロビジョニング可能
  - TTL、パターン削除など豊富な機能
  - Go エコシステムで広く使われている（go-redis/v9）
  - 将来的に Pub/Sub、レートリミットなど他の用途にも活用可能
- **デメリット**
  - 外部依存が増える（運用コスト）
  - ネットワークレイテンシが発生する

### 選択肢 2: インメモリキャッシュ（go-cache / ristretto）

- **メリット**
  - 外部依存なし
  - ネットワークレイテンシなし（ナノ秒オーダー）
  - 導入が簡単
- **デメリット**
  - インスタンス間でキャッシュを共有できない
  - プロセス再起動でキャッシュが消失する
  - Railway でのデプロイごとにキャッシュがリセットされる

### 選択肢 3: HTTP キャッシュ（CDN / Reverse Proxy）

- **メリット**
  - アプリケーションコードの変更が最小限
  - クライアント側でもキャッシュが効く
- **デメリット**
  - 認証付きエンドポイントのキャッシュが困難
  - Signed URL を含むレスポンスのキャッシュ制御が複雑
  - きめ細かいキャッシュ無効化が難しい

## 理由

1. **スケーラビリティ**: Railway でインスタンスを増やした場合にもキャッシュの一貫性を保てる
2. **運用の容易さ**: Railway が Redis をマネージドサービスとして提供しており、プロビジョニングが容易
3. **エコシステム**: go-redis/v9 は Go の Redis クライアントのデファクトスタンダードであり、十分な実績がある
4. **拡張性**: 将来的にキャッシュ以外の用途（レートリミット、セッション管理など）にも活用可能
5. **フォールバック設計**: Redis 障害時に DB にフォールバックすることで、可用性を維持できる

インメモリキャッシュは導入が簡単だが、デプロイのたびにキャッシュがリセットされる点と、将来のスケールアウト時にキャッシュの不整合が発生する点が許容しがたい。

## 結果

- `internal/pkg/cache/` に Redis キャッシュクライアントを実装
- `internal/config/config.go` に `REDIS_URL` 環境変数を追加
- `docker-compose.yml` にローカル開発用 Redis を追加
- DI コンテナに Redis クライアントを登録し、グレースフルシャットダウン時に切断
- Phase 1 として、カテゴリ・ボイスの Repository に Cache-Aside パターンを適用
- Redis 未設定時はキャッシュをスキップし、DB のみで動作する（ローカル開発の利便性確保）
